# Product principles

These principles should guide product decisions, scope choices and design trade-offs.

For **interface** look, feel, and information hierarchy, see [Design principles](design-principles.md) **§11 Information hierarchy**.

The product is **not** tied to a single modelling commission: **project- and layer-specific** claims belong in **catalog metadata** and **layer-scoped** UI, not in generic “about the map” copy.

## 1. Decision support, not decision automation

The product should support ecological reasoning and planning, not replace expert judgement.

It must not imply that:

- low suitability means absence
- high suitability means confirmed presence
- the tool can replace field survey
- the output is a definitive ecological decision

## 2. Interpretability before sophistication

If a feature makes the product more advanced but less understandable, it should be treated sceptically.

Users must be able to understand:

- what the output means
- why a place may score as it does
- how the tool should be used responsibly

## 3. Regional-to-local workflow first

The product should support the natural user journey:

- start with broad patterns
- narrow to a place or site
- inspect local context
- understand drivers of suitability
- use the result in a practical decision

## 4. Ecological trust matters more than technical novelty

A technically clever feature is not useful if users do not trust or understand it.

The product should prioritise:

- ecological plausibility
- transparent framing
- sensible explanations
- clear caveats

## 5. Make limitations visible

Limitations should not be buried in documentation.

The product should make it clear that outputs are:

- decision-support tools
- relative rather than absolute
- useful in context
- best used alongside field knowledge and survey

## 6. Focus on real tasks, not model admiration

The product should be built around what users need to do, not around showcasing the model.

That means prioritising workflows such as:

- survey targeting
- site interpretation
- conservation prioritisation
- communication of evidence

## 7. Keep the first version narrow and useful

The initial product should do a small number of important things well.

Avoid trying to become:

- a broad biodiversity platform
- a generic spatial analytics environment
- a feature-heavy GIS tool
- a technical interface built mainly for modellers

## 8. Support explainable decisions

The product should make it easier for users to justify and communicate why a location or area may matter.

It should help users move from “the map says this” to “here is how we interpret this pattern”.
